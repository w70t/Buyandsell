# خارطة طريق «سوقنا» — Souqna Product Roadmap

> مراجعة شاملة للكود الحالي + الميزات التي ترفع قيمة المنصّة ×10، مقسّمة على مراحل،
> ولكل ميزة **معايير قبول (Acceptance Criteria)** قابلة للاختبار.

---

## 1) ما الموجود اليوم (Baseline)

| المجال | الحالة الحالية |
|---|---|
| الحسابات | تسجيل/دخول بالبريد + JWT مع تحديث توكن، جلسة دائمة |
| الإعلانات | نشر بإعدادات (سعر، حالة، تفاوض، محافظة/مدينة)، صور متعددة، حذف |
| التصفح | رئيسية بأقسام ملوّنة + أحدث الإعلانات مع تحميل تدريجي، شاشة قسم، بحث نصي + فلاتر أساسية |
| المفضلة | إضافة/إزالة مع مزامنة |
| الرسائل | محادثات نصية بين بائع ومشتري (Polling) |
| الإدارة | مسارات admin أساسية |
| البنية | FastAPI + PostgreSQL + Alembic + Nginx + Docker، تخزين ملفات محلي أو MinIO/S3 |
| التصميم | Material 3، RTL، ثيم داكن/فاتح، خط **Cairo** مضمّن، نظام ألوان SxColors |

**الفجوات الكبرى:** لا إشعارات، لا توثيق هوية/ثقة، لا تقييمات، البحث بدائي (بدون فهرسة عربية)،
لا خصائص لكل قسم (سيارة/عقار…)، لا خرائط/موقع دقيق، لا تحقيق دخل، والمحادثة ليست لحظية.

## 2) المرحلة الأولى — الثقة والتفاعل (أسابيع 1–4)

أعلى أثر مقابل الجهد؛ بدونها لا يعود المستخدم.

### F1. إشعارات فورية (Push Notifications — FCM)
لماذا: الرسالة التي لا يعرف بها البائع = صفقة ميتة. أكبر رافعة احتفاظ (Retention) متاحة.

**معايير القبول:**
- [ ] عند استلام رسالة جديدة والتطبيق مغلق، يصل إشعار خلال ≤ 10 ثوانٍ يفتح شاشة المحادثة مباشرة.
- [ ] إشعار عند إضافة إعلانك للمفضلة، وعند تغيّر حالة إعلانك (موافقة/رفض إداري).
- [ ] جدول `device_tokens` في القاعدة مع تنظيف التوكنات الميتة (استجابة FCM غير الصالحة تحذف التوكن).
- [ ] شاشة إعدادات تتيح تفعيل/تعطيل كل نوع إشعار على حدة، وتُحترم الخيارات في الخادم.
- [ ] فتح الإشعار يمرّ عبر Deep Link موحّد (`souqna://listing/{id}` و`souqna://chat/{id}`).

### F2. توثيق برقم الهاتف (OTP) + شارة موثّق
لماذا: في السوق العراقي الهاتف هو الهوية؛ التوثيق يقطع أغلب الحسابات الوهمية.

**معايير القبول:**
- [ ] التسجيل/الدخول برقم هاتف عراقي (+964) عبر رمز OTP (مزوّد SMS أو WhatsApp Business API).
- [ ] معدّل محاولات مقيّد: ≤ 3 رموز لكل رقم في الساعة، والرمز صالح 5 دقائق فقط.
- [ ] شارة «موثّق ✓» تظهر على بطاقة الإعلان وصفحة البائع لمن وثّق رقمه.
- [ ] الحسابات البريدية القديمة تعمل كما هي، مع مسار ترقية لإضافة الهاتف.

### F3. تقييمات البائعين (Ratings & Reviews)
لماذا: الثقة هي المنتج الحقيقي لأي سوق مستعمل.

**معايير القبول:**
- [ ] بعد محادثة فعلية فقط (يوجد سجل رسائل بين الطرفين) يمكن ترك تقييم 1–5 نجوم + تعليق اختياري.
- [ ] تقييم واحد لكل مشترٍ لكل إعلان؛ يمكن تعديله خلال 48 ساعة.
- [ ] متوسط التقييم وعدد المقيّمين يظهران في صفحة البائع وبطاقة الإعلان التفصيلية.
- [ ] لا يستطيع المستخدم تقييم نفسه، وتُخفى التقييمات المُبلَّغ عنها بعد مراجعة إدارية.

### F4. الإبلاغ والإشراف (Reporting & Moderation) — ✅ منجز
**معايير القبول:**
- [x] زر «إبلاغ» على كل إعلان وصفحة بائع مع أسباب مصنّفة (احتيال، محتوى مخالف، سلعة ممنوعة…).
- [x] لوحة إدارية تعرض البلاغات مرتّبة بالأحدث مع إجراءات: إخفاء الإعلان، حذف الإعلان، حظر المستخدم، رفض البلاغ.
- [x] الإعلان الذي يتجاوز N بلاغات (`REPORTS_AUTO_HIDE_THRESHOLD`) يُخفى تلقائياً بانتظار المراجعة.
- [x] كل إجراء إداري مسجَّل بسجل تدقيق (من، متى، ماذا) — `/admin/audit`.

> **منجز أيضاً خارج الترقيم:** إشعارات داخل المنصّة (رسالة/مفضلة/إشراف) مع عدّاد
> غير المقروء، إشعارات تيليغرام للمشرف، موقع ويب عام SSR بالعربية، ولوحة إدارة ويب
> كاملة — انظر `TASKS.md`.

## 3) المرحلة الثانية — اكتشاف أذكى (أسابيع 5–9)

### F5. بحث عربي حقيقي + فرز متقدم
لماذا: البحث هو الواجهة الأولى للمشتري الجاد.

**معايير القبول:**
- [ ] فهرسة PostgreSQL FTS + `pg_trgm` مع تطبيع عربي (همزات/تاء مربوطة/ياء): البحث عن «تلفون» يعيد «تليفون».
- [ ] فرز بالنتائج: الأحدث، الأرخص، الأغلى، الأقرب (بعد F7).
- [ ] فلترة مركّبة تعمل معاً: قسم + محافظة + مدى سعر + حالة السلعة.
- [ ] زمن استجابة البحث p95 ≤ 300ms على 100k إعلان (اختبار حمل موثّق).
- [ ] كلمات البحث الشائعة تُسجَّل (مجهولة الهوية) لتغذية الاقتراحات.

### F6. خصائص ديناميكية لكل قسم (Category Attributes)
لماذا: «سيارة» بدون (النوع/الموديل/سنة/ممشى) إعلان نصف ميت — هذه أكبر قفزة جودة للمحتوى.

**معايير القبول:**
- [ ] مخطط `attributes` (JSONB + تعريف لكل قسم): سيارات (ماركة، موديل، سنة، ممشى، وقود، جير)، عقارات (نوع، غرف، مساحة م²، طابق)، هواتف (ماركة، ذاكرة، حالة الشاشة).
- [ ] نموذج نشر الإعلان يعرض حقول القسم المختار ديناميكياً مع تحقق (سنة ضمن نطاق، أرقام فقط…).
- [ ] الفلاتر في البحث تشمل خصائص القسم (مثال: كل تويوتا 2015–2020 في بغداد تحت 15 مليون).
- [ ] الخصائص تظهر في صفحة الإعلان كجدول مرتب وليس نصاً حراً.

### F7. الموقع الجغرافي والخريطة
**معايير القبول:**
- [ ] عند النشر يمكن تثبيت موقع تقريبي على الخريطة (لا يُعرض الموقع الدقيق للعامة — دائرة تقريبية).
- [ ] فرز «الأقرب إليّ» يعمل بإذن الموقع، ويتدهور بأمان إلى فرز المحافظة عند رفض الإذن.
- [ ] عرض «خريطة النتائج» للبحث مع تجميع (Clustering) عند التصغير.
- [ ] الاستعلام الجغرافي عبر PostGIS أو حساب Haversine مفهرس؛ p95 ≤ 400ms.

### F8. عمليات بحث محفوظة + تنبيهات
**معايير القبول:**
- [ ] «احفظ هذا البحث» من شاشة النتائج بكل فلاتره.
- [ ] عند نشر إعلان جديد يطابق بحثاً محفوظاً، يصل إشعار خلال ≤ 5 دقائق.
- [ ] حدّ أقصى 10 عمليات محفوظة لكل مستخدم مع إدارة (تعديل/حذف/إيقاف مؤقت).

## 4) المرحلة الثالثة — تحقيق الدخل والنمو (أسابيع 10–14)

### F9. الإعلانات المميّزة (Featured Listings) + دفع محلي
لماذا: أول مصدر دخل طبيعي للمنصّات المبوّبة، بلا مساس بمجانية الأساس.

**معايير القبول:**
- [ ] باقات ترويج: «مميز 7 أيام» يظهر في شريط أعلى الرئيسية وبشارة ملوّنة في النتائج.
- [ ] تكامل دفع محلي (ZainCash / آسيا حوالة / بطاقة) عبر طبقة `PaymentProvider` قابلة للتبديل.
- [ ] عملية الدفع لها حالات صريحة (pending/paid/failed/refunded) وWebhook موقَّع من المزوّد.
- [ ] انتهاء الترويج تلقائي (مهمة مجدولة) ويعود الإعلان عادياً دون تدخل.
- [ ] فواتير/إيصالات داخل التطبيق، ولوحة إدارية بإجمالي الإيرادات يومياً/شهرياً.

### F10. محادثة لحظية (WebSocket) + وسائط
**معايير القبول:**
- [ ] استبدال الـ Polling بـ WebSocket: وصول الرسالة للطرف الآخر ≤ 1 ثانية والتطبيق مفتوح.
- [ ] مؤشرات: «يكتب الآن…»، صحّ واحدة (أُرسلت) / صحّتان (قُرئت).
- [ ] إرسال صور داخل المحادثة (بحد حجم وضغط تلقائي).
- [ ] عرض «سعر مقترح» كرسالة منظّمة (عرض/قبول/رفض) مرتبطة بالإعلان.
- [ ] عند انقطاع الاتصال يعاد الاتصال تلقائياً مع جلب الفائت (لا رسائل ضائعة — اختبار قطع شبكة موثّق).

### F11. تحسين الصور والأداء
**معايير القبول:**
- [ ] توليد أحجام متعددة (thumbnail/medium/full) بصيغة WebP عند الرفع؛ البطاقات تحمّل thumbnail فقط.
- [ ] BlurHash/ThumbHash لكل صورة — لا مربعات رمادية أثناء التحميل.
- [ ] حجم صفحة الرئيسية الأولى (شبكة 20 إعلاناً) ≤ 500KB شبكياً.
- [ ] فحص أن الصورة صورة فعلاً (MIME + أبعاد) ورفض الملفات الضارة.

### F12. لوحة تحليلات إدارية
**معايير القبول:**
- [ ] مؤشرات يومية: مستخدمون جدد، إعلانات جديدة، رسائل، DAU/WAU، نسبة إعلانات بصور.
- [ ] أكثر الأقسام والمحافظات نشاطاً (رسوم بيانية).
- [ ] تصدير CSV لأي جدول مؤشرات.

## 5) المرحلة الرابعة — التميّز بالذكاء (أسابيع 15+)

### F13. مساعد نشر ذكي (AI Listing Assistant)
لماذا: أكبر احتكاك في المنصّة هو كتابة إعلان جيد — الذكاء الاصطناعي يحوّله لثلاث ثوانٍ.

**معايير القبول:**
- [ ] التقاط صورة ⇠ اقتراح تلقائي: القسم + عنوان + وصف عربي قابل للتعديل قبل النشر.
- [ ] «سعر السوق المقترح» بناء على إعلانات مشابهة منشورة (مدى سعري وليس رقماً واحداً).
- [ ] كشف تلقائي للمحتوى المخالف في الصور قبل النشر (تصنيف + مراجعة بشرية للحالات الرمادية).
- [ ] كل اقتراح قابل للرفض بنقرة، ولا يُنشر شيء دون تأكيد المستخدم.

### F14. توصيات «مشابه لهذا» و«مختارات لك»
**معايير القبول:**
- [ ] صفحة الإعلان تعرض 6 إعلانات مشابهة (نفس القسم + قرب سعري/جغرافي) بنسبة نقر ≥ 5%.
- [ ] قسم «مختارات لك» في الرئيسية مبني على تصفّح ومفضلة المستخدم (وبدون تسجيل: الأكثر تفاعلاً في محافظته).

### F15. نسخة ويب عامة (SEO)
**معايير القبول:**
- [ ] كل إعلان له صفحة ويب عامة بعنوان URL ثابت وبيانات Schema.org (`Product`/`Offer`).
- [ ] صفحات الأقسام والمحافظات قابلة للفهرسة (SSR أو Prerender) — ظهور في نتائج بحث Google خلال شهر.
- [ ] زر «افتح في التطبيق» ذكي (App Links / Universal Links).

## 7) مؤشرات النجاح (North-Star KPIs)

- **وقت أول رد للبائع** ≤ 30 دقيقة (بعد F1/F10).
- **نسبة الإعلانات المكتملة** (صور + خصائص قسم) ≥ 80% (بعد F6/F13).
- **احتفاظ الأسبوع الرابع (W4 Retention)** ≥ 25% (بعد المرحلة 1–2).
- **إيراد شهري متكرر** من الترويج يغطي كلفة البنية (بعد F9).
